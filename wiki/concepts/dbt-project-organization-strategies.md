# dbt Project Organization Strategies

The set of decisions around repository structure (monorepo vs. multirepo) and project structure (mono-project vs. multi-project) for dbt implementations. These decisions are tightly linked and scale with team size and organizational complexity.

## Key Decisions

### Repository Structure
- **Monorepo**: Single repository containing all dbt projects and related code. Recommended for small teams and organizations just starting with dbt.
- **Multirepo**: Multiple repositories, one per domain or project. Suitable for larger organizations with clear domain boundaries.

### Project Structure
- **Mono-project**: Single dbt project (generated by `dbt init`). The default and recommended starting point.
- **Multi-project**: Multiple dbt projects, typically organized by domain. Requires cross-project interfaces.

### Cross-Project Interfaces
- **Exposures**: Define downstream usage of models across projects. Preferred approach for multi-project setups. Exposures in one project can be automated to generate sources in another project.
- **Packages**: Install projects as dependencies. Can lead to dependency management complexity and potential duplicate model execution.

### Model Folder Structure
- **dbt recommended structure**: Follows dbt Labs' best practices guide.
- **Brice's recommendations**: Alternative approach focusing on end-user perspective.
- **General advice**: Don't be shy to create folders to separate concerns.

### YAML Organization
- **One YAML per model**: Recommended approach for model descriptions.
- **Sources at schema/database level**: Sources should be organized at the schema or database level.

## Recommendations

- Start with one repo, one project for small teams.
- Use exposures for cross-project interfaces rather than packages.
- Organize models with clear folder separation.
- Keep one YAML per SQL file for models.
- dbt repo should be independent from other tools (e.g., not in the Airflow repo).
