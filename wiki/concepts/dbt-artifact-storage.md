# dbt Artifact Storage

**dbt Artifact Storage** is the mechanism by which dbt Core–generated JSON files (`manifest.json`, `catalog.json`, `run_results.json`) are made accessible to OpenMetadata for metadata ingestion. For **dbt Core** deployments, OpenMetadata cannot directly access the local filesystem where dbt executes (e.g., Airflow, Kubernetes, GitHub Actions). The artifacts must be stored in a supported cloud storage service or shared filesystem. This is a **required** step for dbt Core users; **dbt Cloud** users bypass this entirely by using the Cloud API to retrieve metadata directly.

## Required dbt Artifacts

| File | Generated By | Purpose |
|------|-------------|---------|
| `manifest.json` | `dbt run`, `dbt compile`, `dbt build` | Required — Model definitions, sources, lineage, tests |
| `catalog.json` | `dbt docs generate` | Recommended — Column names, types, descriptions |
| `run_results.json` | `dbt test`, `dbt run`, `dbt build` | Optional — Test execution results and timing |

These files are stored in the `target/` directory of your dbt project after each dbt execution.

## Supported Storage Backends

| Backend | Typical Use Case |
|---------|-----------------|
| **Amazon S3** | AWS deployments (ECS, EKS, EC2) |
| **Google Cloud Storage (GCS)** | GCP deployments (GKE, Compute) |
| **Azure Blob Storage** | Azure deployments (AKS, VMs) |
| **HTTP/HTTPS Server** | Cloud-agnostic, static file servers |
| **Local/Shared Filesystem** | Single-server or Docker deployments |

Each backend requires appropriate permissions and configuration to allow OpenMetadata to read the artifacts.

## Configuration Steps

1. **Generate artifacts**: Run the dbt commands in the correct order (see [dbt Command Sequence](#dbt-command-sequence) below) to produce `manifest.json`, `catalog.json`, and `run_results.json`.
2. **Choose storage method**: Select the backend appropriate for your infrastructure (e.g., S3, GCS, Azure, HTTP, or local filesystem).
3. **Upload artifacts**: Configure your dbt workflow to upload artifacts to the chosen storage location after each dbt execution. Ensure the upload happens only after all dbt commands complete.
4. **Configure OpenMetadata**: Provide the storage path and credentials during ingestion setup in the OpenMetadata UI.

### dbt Command Sequence

Run the following commands in order to generate all artifacts correctly:

```bash
dbt run              # Execute models, generates manifest.json
dbt test             # Run tests, updates run_results.json
dbt docs generate    # Generate catalog.json with column metadata
```

Incorrect order can lead to missing or stale artifacts.

## dbt Cloud

dbt Cloud users do not need to configure artifact storage. OpenMetadata retrieves metadata directly from dbt Cloud APIs, eliminating the requirement for external cloud storage configuration. See [[dbt-integration]] for the distinction between Core and Cloud integration models.

## Key Requirement for Lineage

The `compiled_code` field in `manifest.json` is **required** for lineage extraction. If lineage is missing, ensure `dbt compile` or `dbt docs generate` has been run before uploading artifacts. See [[dbt-lineage-ingestion]] for details.

## Troubleshooting Common Issues

| Issue | Possible Cause | Solution |
|-------|---------------|----------|
| Artifacts not found | Upload failed or wrong path | Verify artifacts uploaded successfully to correct location |
| Access denied | Insufficient permissions | Check IAM policies, service account permissions, or access keys |
| Stale metadata | Old artifacts | Ensure upload happens after dbt completes; verify timestamps |
| Missing lineage | No `compiled_code` in manifest | Run `dbt compile` or `dbt docs generate` before upload |
| Missing column descriptions | No catalog.json | Ensure `dbt docs generate` runs and catalog.json is uploaded |
| Test results not showing | No run_results.json | Ensure `dbt test` runs and run_results.json is uploaded |
