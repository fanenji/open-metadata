# LLM SQL Generation Evaluation

Approaches to evaluating the quality of SQL generated by Large Language Models (LLMs) and Small Language Models (SLMs). Evaluation methods vary in strictness and focus.

## Evaluation Approaches

### Execution Match
Comparing the results of executing the generated SQL against a reference SQL. The [[semantic-sql-testing-benchmark]] uses **exact result matching**, requiring 100% row match per column with 0.5% numeric tolerance for floating point aggregations. This is stricter than execution-based metrics that only check for correct row counts or aggregate values.

### Text Match
Comparing the generated SQL text against a reference SQL using string similarity metrics (e.g., BLEU, ROUGE, exact string match). This approach is common in benchmarks like Spider but can penalize semantically equivalent but syntactically different queries.

### Semantic Match
Evaluating whether the generated SQL captures the correct business logic and intent, independent of syntactic form. This is the most challenging approach and often requires human judgment or a semantic model.

## Key Dimensions

- **Strictness** — Exact result matching is the strictest; text matching is moderate; execution-based matching (e.g., checking row count only) is the least strict
- **Feedback loop** — Some evaluations measure a model's ability to self-correct after seeing initial results, as in the semantic SQL testing benchmark
- **Reproducibility** — The choice of reference baseline generator (e.g., Claude Opus) affects reproducibility
