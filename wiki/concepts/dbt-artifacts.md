# dbt Artifacts

dbt automatically generates JSON files — called artifacts — in the `target/` directory whenever you run commands like `dbt run`, `dbt test`, or `dbt docs generate`. These artifacts contain metadata about dbt models, sources, tests, and execution results. OpenMetadata reads these pre-generated files to import metadata about dbt transformations. They are the foundation of the [[dbt-integration]] with OpenMetadata, providing all the metadata needed to build lineage graphs, sync documentation, track data quality, and import governance metadata.

## Artifact Files

| File | Generated By | Purpose |
|------|-------------|---------|
| manifest.json | `dbt run`, `dbt compile`, `dbt build` | **Required** — Contains model definitions, sources, lineage information, test configurations, and the `compiled_code` field essential for lineage extraction. |
| catalog.json | `dbt docs generate` | **Recommended** — Contains column names, types, and descriptions. Without this file, column-level metadata will be missing. |
| run_results.json | `dbt test`, `dbt run`, `dbt build` | **Optional** — Contains test execution results and timing information. Without this file, test results will not appear in OpenMetadata. |

### manifest.json

The manifest is the heart of dbt metadata. **This file is mandatory** for the integration to work.

**Contents:**
- Model definitions and SQL code
- `ref()` and `source()` dependencies for lineage
- Descriptions from `schema.yml` files
- dbt tags and meta properties
- Test configurations
- Column definitions

**OpenMetadata usage:**
- Creates Data Model entities linked to database tables
- Builds lineage graphs showing data flow between models
- Syncs table and column descriptions
- Creates classification tags
- Assigns ownership and domains
- Creates test cases for data quality monitoring

**Important:** The `compiled_code` field in manifest.json is critical for lineage extraction. Run `dbt compile` or `dbt docs generate` to ensure it is present.

**Generated by:** Any dbt command (`dbt run`, `dbt build`, `dbt compile`)

### catalog.json

The catalog provides database-level details not present in the manifest.

**Contents:**
- Actual column data types from the database
- Database-level ownership information
- Column ordering as it exists in the database
- Statistics about tables and columns

**OpenMetadata usage:**
- Provides accurate column data types (more reliable than `schema.yml` declarations)
- Fallback owner information if not specified in meta properties
- Maintains column position from the database

**Generated by:** `dbt docs generate`

Without catalog.json, lineage and model information are still available, but column types will only reflect what is declared in `schema.yml` files. Column names, types, and descriptions will be unavailable.

### run_results.json

Run results capture the outcome of the most recent dbt execution.

**Contents:**
- Test pass/fail/warn status for each test
- Execution timestamps
- Error messages and stack traces
- Model build success/failure status

**OpenMetadata usage:**
- Updates test case results showing pass/fail status
- Tracks when tests last ran
- Shows failure details for debugging

**Generated by:** `dbt run`, `dbt test`, `dbt build`

Run `dbt test` before the OpenMetadata ingestion runs to capture the latest test results.

## Generation Workflow

Run these commands in order to generate all artifacts:

```shell
# Step 1: Run models (generates manifest.json)
dbt run

# Step 2: Run tests (updates run_results.json with test outcomes)
dbt test

# Step 3: Generate catalog (creates catalog.json)
dbt docs generate
```

Verify artifacts exist:

```shell
ls -la target/*.json
# Expected: target/manifest.json, target/catalog.json, target/run_results.json
```

## Critical Notes

- **manifest.json is required** — Without it, OpenMetadata cannot ingest any dbt metadata.
- **catalog.json is recommended** — Without it, column names, types, and descriptions are missing; lineage and model information are still available.
- **run_results.json is optional** — Without it, test execution results and timing are unavailable.
- **compiled_code is critical for lineage** — The `compiled_code` field in manifest.json must be present for lineage extraction. Run `dbt compile` or `dbt docs generate` to ensure it exists.
- **Artifact freshness** — Stale artifacts cause outdated metadata. Ensure uploads happen immediately after dbt completes.

## Storage Configuration

For dbt Core users, artifacts must be uploaded to a storage backend accessible by OpenMetadata. See [[dbt-artifact-storage]] for the five supported backends and configuration details. dbt Cloud users do not need storage configuration — see [[dbt-core-vs-cloud-storage]].
