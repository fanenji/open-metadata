# Dremio Reflections

**Reflections** are Dremio's unique acceleration layer — materialized views stored as Apache Iceberg tables, combined with an intelligent caching system that pre-computes and stores frequently accessed data for sub‑second query responses. When a query is submitted, Dremio automatically routes it to the most efficient reflection if available, transparently to the user or application, without requiring any changes to SQL or BI tools.

## Key Characteristics

- **Automatic and Transparent Optimization** — Dremio's query planner automatically uses reflections when beneficial, and queries are accelerated without user intervention or query modification.
- **Open and Portable** — Reflections are stored as Apache Iceberg tables, leveraging the open table format and avoiding vendor lock-in. Unlike traditional BI cubes or extracts, Reflections benefit any team using the same data in any tool.
- **Low Maintenance** — Reflections are refreshed on a cadence to reflect the most up-to-date information. Dremio also provides reflection recommendations to identify optimization opportunities. In Dremio Cloud, reflections can be managed automatically with autonomous caching.
- **Multiple Reflection Types** — Raw, aggregation, and partition reflections provide flexibility for different use cases (see [Types of Reflections](#types-of-reflections)).
- **Managed via UI, API, or Programmatically** — Reflections can be created and managed through Dremio's user interface, REST API, or programmatically using [[dremioframe]].

## Architectural Role

Reflections serve as the primary performance optimization mechanism in the Dremio platform. They enable sub‑second query responses on large datasets without requiring Apache Spark or other external processing engines — making Reflections a key reason some deployments choose not to implement Spark separately. By pre-computing and caching data from underlying sources (MinIO/Iceberg, Oracle, etc.), reflections accelerate all downstream consumers.

In a typical deployment, a Dremio cluster (e.g., 1 coordinator + 3 executors with 128 GB RAM each) provides substantial memory for reflections, enabling acceleration of the most critical query patterns.

## Types of Reflections

- **Raw Reflections** — A full columnar cache of raw data. Useful for accelerating column pruning and filtering on wide tables.
- **Aggregation Reflections** — Pre-computed aggregates on selected dimensions and measures (e.g., GROUP BY, SUM, COUNT). Ideal for accelerating BI dashboard queries.
- **Partition Reflections** — A partitioned layout of the data organized by partition key, which can accelerate queries that filter on those keys.

## Management and Refresh

Reflections are refreshed on a cadence to keep data up‑to‑date. Dremio provides reflection recommendations to help identify optimization opportunities. In Dremio Cloud, reflections benefit from autonomous caching, reducing the need for manual configuration.

Management can be performed through:
- **Dremio UI** – Create and manage reflections via the graphical interface.
- **REST API** – Automate reflection management through Dremio's API.
- **Programmatic access** – Using [[dremioframe]] to create, list, or manage reflections.

## Use Cases and Examples

- **Accelerating BI Dashboards** – Reflections pre‑compute common aggregations, reducing query latency for BI tools like [[apache-superset]] and Power BI.
- **Pre‑computing Aggregations for Reporting** – Common aggregate queries are served from a pre‑computed reflection instead of scanning the entire table.
- **Reducing Query Latency for Federated Queries** – By caching data from multiple sources as a reflection, cross‑source queries are faster.
- **Optimizing Iceberg Table Queries** – Reflections optimize queries on Iceberg tables without manual tuning.
- **Efficient [[dbt]] Transformation Execution** – Reflections accelerate the SQL queries generated by dbt, reducing transformation run times.
- **Sub‑second Query Response on Large Datasets** – By leveraging pre‑computed reflections, even large‑scale datasets can be queried interactively without Spark or other heavy engines.

### Hands‑On Guide Example

In the practical guide, Reflections are enabled on production views by selecting dimensions and measures to optimize. Once created, Dremio uses them to accelerate aggregate queries on those dimensions and measures.

## Relationship to Other Concepts

- **[[dbt]] Materializations** – dbt manages the semantic layer (views and tables), while Reflections accelerate queries on those datasets. They are complementary rather than competitive.
- **Traditional BI Cubes/Extracts** – Unlike tool‑specific cubes or extracts, Reflections are vendor‑neutral and accessible to any query engine that can read Iceberg tables.
- **Iceberg Table Versioning** – Reflections are acceleration structures, not data versioning mechanisms.
- **DuckDB Caching** – Reflections are server‑side acceleration, while DuckDB caching is client‑side.
- **[[dremio]] Performance Toolkit** – Reflections are part of Dremio's optimization features alongside the columnar cloud cache.

Reflections are Dremio's answer to materialized views in traditional data warehouses, but built on Iceberg for portability and augmented with in‑memory caching for speed.
