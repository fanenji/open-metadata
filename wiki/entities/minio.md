# MinIO

**MinIO** is a high-performance, distributed, S3-compatible object storage system that serves as the central storage layer and physical core of the [[data-lakehouse]] in the Regione Liguria Data Platform. It provides the distributed storage foundation for compute engines like [[dremio]] and Spark, and serves as the primary storage layer for Cloud Optimized GeoTIFF (COG) files in the raster data pipeline. By offering full S3 API compatibility, MinIO enables seamless integration with tools designed for Amazon S3 while maintaining complete control over the on‑premise infrastructure.

## Key Features

- **S3 API Compatibility**: Full compatibility with the Amazon S3 API for broad tool integration.
- **Erasure Coding**: Data redundancy that protects against node and drive failures.
- **Bit Rot Protection**: Data integrity against silent corruption.
- **Object Immutability**: Write‑once, read‑many (WORM) support.
- **Bucket Notifications**: Event‑driven triggers for downstream processing.
- **Versioning**: Object‑level versioning for data recovery.
- **Data Tiering**: Automatic movement of data between storage tiers based on access patterns.
- **Scalability**: Designed to handle massive amounts of unstructured and structured data.
- **Security**: LDAP/OpenID authentication integration.

## Deployment Specifications

The recommended production deployment consists of **4 nodes** with the following resources per node:

| Component | Specification (primary) | Alternative specification |
|-----------|--------------------------|---------------------------|
| vCPU      | 16                       | 16                        |
| RAM       | 32 GB                    | 32 GB                     |
| Storage   | 4 × 4 TB NVMe SSDs (16 TB raw per node) | 4 × 1 TB flash storage (4 TB raw per node) |

- **Total raw storage**: 64 TB (primary) or 16 TB (alternative).
- **Authentication**: LDAP/OpenID integration.

### Erasure Coding Configuration

The erasure coding configuration uses a stripe size of **16** and a parity of **8**:

| Metric | Value (based on 64 TB raw) |
|--------|----------------------------|
| **Usable capacity** | ~32 TiB (≈32 TB) |
| **Storage efficiency** | 50% |
| **Drive failure tolerance** | 8 drives out of 16 |
| **Server failure tolerance** | 2 servers out of 4 |

This configuration represents a deliberate trade‑off between storage efficiency and fault tolerance. The 50% efficiency is critical for capacity planning: for every 2 TB of raw storage, only 1 TB is usable. Dremio executors require 300–500 GB of distributed storage from MinIO, which must be accounted for in the usable capacity calculation.

## Role in Architecture

### Data Lakehouse
MinIO acts as the foundation for the Data Lakehouse, hosting:
- **Raw Data**: Incoming streams from Kafka and batch loads (Bronze layer).
- **Processed Data**: Structured datasets in [[apache-iceberg]] format (Silver/Gold layers).

All raw data ingested by [[kestra]] is first loaded into MinIO as the Bronze layer of the [[bronze-silver-gold-architecture]]. Apache Iceberg tables are stored on MinIO, and [[project-nessie]] catalogs reference MinIO paths for versioning and branching.

### Raster Data Pipeline
MinIO provides the scalable, cloud‑native storage backend for raster data, distinct from the Iceberg/Nessie catalog used for vector data:
- **COG Storage**: Stores Cloud Optimized GeoTIFF files generated by the Geoscript/Python+GDAL pipeline.
- **S3 Path Access**: COG files are accessed directly via their S3 path (e.g., `s3://bucket-name/raster/...`), enabling streaming access from QGIS, Python with Rasterio, and other GIS clients.
- **Directory Organization**: The S3 prefix structure (e.g., `s3://bucket/raster/ortofoto/anno=2023/`) is used for logical organization and cataloging.

## Integration with Other Components

- **Data Catalog (DataHub/OpenMetadata)**: The S3 base path is stored as metadata in the catalog, enabling users to discover and access COG files.
- **STAC Server**: The S3 path serves as the `href` for STAC assets, linking the catalog entry to the actual data.
- **Dremio & Spark**: Provide compute over data stored in MinIO. For vector/Iceberg data, Dremio queries the Iceberg tables via Nessie catalog. For raster data, direct S3 access is recommended rather than through Dremio.
- **Kestra**: Directs raw data ingestion to MinIO as the initial storage layer.
- **Project Nessie**: References MinIO paths in its catalogs for Iceberg table versioning.
