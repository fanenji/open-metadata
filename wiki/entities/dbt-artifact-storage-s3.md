# dbt Artifact Storage S3

AWS S3 is a supported storage backend for dbt Core artifacts in OpenMetadata v1.12.x. This page documents the complete configuration workflow for using S3 as the intermediary storage layer between dbt execution and OpenMetadata ingestion.

## Overview

Using S3 as the artifact storage backend decouples dbt execution from OpenMetadata. The dbt artifacts (manifest.json, catalog.json, run_results.json) are generated by dbt, uploaded to S3 by an orchestrator like Airflow, and then consumed by OpenMetadata from S3. No OpenMetadata packages need to be installed in the Airflow environment.

## IAM Policy Separation

A key security best practice is the use of two separate IAM policies following the least-privilege principle:

### Write Policy (for dbt/Airflow)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowDBTArtifactUpload",
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:PutObjectAcl"],
      "Resource": "arn:aws:s3:::your-company-dbt-artifacts/dbt-artifacts/*"
    },
    {
      "Sid": "AllowBucketListing",
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::your-company-dbt-artifacts"
    }
  ]
}
```

### Read Policy (for OpenMetadata)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowOpenMetadataRead",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::your-company-dbt-artifacts",
        "arn:aws:s3:::your-company-dbt-artifacts/dbt-artifacts/*"
      ]
    }
  ]
}
```

## Credential Options

OpenMetadata supports two authentication methods for S3 access:

- **Option A: Access Keys** — Universal method using AWS Access Key ID and Secret Access Key. Works regardless of where OpenMetadata is deployed.
- **Option B: IAM Role** — When OpenMetadata runs on AWS infrastructure (e.g., EKS), the IAM role can be used by leaving the Access Key fields empty. The exact IAM trust policy requirements are not fully documented.

## Airflow DAG Pattern

The reference implementation includes a complete Airflow DAG that:
1. Runs dbt models (`dbt run`)
2. Runs dbt tests (`dbt test`)
3. Generates dbt documentation (`dbt docs generate`)
4. Uploads all artifacts to S3 using a PythonOperator with boto3

The DAG uses environment variables for configuration (DBT_PROJECT_DIR, S3_BUCKET, S3_PREFIX) and includes error handling for required vs. optional artifacts.

## OpenMetadata UI Configuration

In the OpenMetadata UI, the S3 source is configured under the database service's Ingestion tab:
- **dbt Configuration Source**: S3
- **S3 Bucket Name**: your-company-dbt-artifacts
- **S3 Object Prefix**: dbt-artifacts
- **AWS Region**: us-east-1
- **AWS Credentials**: Access Keys or IAM Role
- **dbt Options**: Update Descriptions, Update Owners, Include Tags, Classification Name

## Troubleshooting

Common issues include:
- **Access Denied (403)**: IAM permissions insufficient — verify s3:GetObject and s3:ListBucket
- **Manifest not found**: S3 path incorrect — check dbtObjectPrefix matches S3 structure
- **No lineage**: Database metadata not ingested first — run database metadata ingestion before dbt ingestion
- **Stale data**: Old artifacts in S3 — verify dbt DAG uploads fresh artifacts
- **Missing columns**: Missing catalog.json — ensure dbt docs generate runs and uploads
