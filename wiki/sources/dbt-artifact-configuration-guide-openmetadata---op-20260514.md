# dbt Artifact Configuration Guide | OpenMetadata

This is the official OpenMetadata documentation page for configuring dbt Core artifact storage. It explains the critical distinction between dbt Core (requires storage configuration) and dbt Cloud (no storage needed, uses Cloud API directly). The guide covers the three-artifact model (manifest.json, catalog.json, run_results.json), the required dbt command sequence, five supported storage backends (AWS S3, GCS, Azure Blob, HTTP Server, Local/Shared Filesystem), and a troubleshooting table for common issues.

## Key Content

- **dbt Core vs. dbt Cloud**: dbt Core users MUST configure artifact storage; dbt Cloud users skip this entirely.
- **Three-Artifact Model**: manifest.json (required), catalog.json (recommended), run_results.json (optional).
- **dbt Command Sequence**: `dbt run` → `dbt test` → `dbt docs generate` — incorrect order causes missing artifacts.
- **Storage Backends**: Five options with CLI commands for quick setup: S3, GCS, Azure, HTTP, Local.
- **Troubleshooting**: Maps symptoms (artifacts not found, access denied, stale metadata, missing lineage, missing column descriptions, test results not showing) to causes and solutions.
- **compiled_code Requirement**: Explicitly states that `compiled_code` in manifest.json is required for lineage.
