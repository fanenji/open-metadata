Agent Skills teach it how to drive.Abhishek Kumar GuptaFollow11 min read·Apr 27, 2026621ListenShareMorePress enter or click to view image in full sizeEvery data engineer who has used an AI coding assistant on a dbt project has hit the same wall.You ask the agent to build a new staging model. It does something. The SQL compiles. You feel optimistic. Then you look closely and realize it built all five models at once, declared victory, never tested anything, never validated against the DAG, and used a syntax that was deprecated eight months ago.The agent was not dumb. It was uninformed. It had no playbook for how dbt work actually gets done — iteratively, with tests, with upstream context, with governance checks at every step.dbt Agent Skills are the answer to exactly this problem.What Are dbt Agent Skills?dbt Agent Skills are a curated, open-source collection of structured instructions that tell AI coding agents how to behave when working inside a dbt project.Think of the relationship this way:MCP gives the agent access to tools and governed context — your models, metrics, lineage, and CLIAgent Skills give the agent the playbook for how to use those tools correctlyWithout Agent Skills, an AI agent with MCP access is like handing a skilled but inexperienced intern the keys to your entire data platform. They can technically do anything. But they do not know your team’s standards, your testing requirements, your iteration patterns, or the subtle gotchas that every experienced dbt practitioner has learned the hard way.Agent Skills encode that institutional knowledge. They represent dozens of hours of focused work by dbt experts, backed by years of real-world dbt usage, made available as structured context that any AI agent can load before starting work.Press enter or click to view image in full sizeHow dbt Agent Skills transform generalist coding agents into highly capable data agents. Source: dbt LabsWhy This Matters: The Benchmark Storydbt Labs ran a benchmark to measure the actual impact of Agent Skills on agent performance. The headline numbers are modest but the detail tells a more important story.Without Agent Skills, the benchmark accuracy rate was 56%. With Agent Skills, it rose to 58.5%.That 2.5 percentage point gain might sound underwhelming. But here is the part that matters: certain complex tasks went from a 0% success rate to reliable completion.The biggest gains were on iterative tasks that require working step-by-step across a dbt DAG — which is exactly the category of work most data engineers actually do.The benchmark revealed a concrete example of this. When asked to produce multiple models based on their schema.yml definitions:Without Skills: The agent created all 6 models at once and declared victory. No validation. No iteration. No tests.With Skills: The agent worked model by model, validated each step, and completed the task successfully every time.That is not a small improvement. In production data engineering, an agent that builds five models in one shot without testing is not helpful — it is a liability. An agent that works iteratively with validation is something you can actually trust.The Full Skill Librarydbt Agent Skills are organized into four major categories.1. Analytics Engineering SkillsThese are the core dbt development skills every data engineer needs an agent to understand.Press enter or click to view image in full sizeThe using-dbt-for-analytics-engineering skill is the most foundational. It encodes the core iterative workflow that experienced dbt practitioners follow: explore upstream context first, build incrementally, validate at each step, never ship without tests.2. Semantic Layer SkillsThese skills teach agents to work correctly with MetricFlow and the dbt Semantic Layer — an area where agents are particularly prone to errors because the syntax changes faster than AI training data.Press enter or click to view image in full sizeThe Semantic Layer skill solves a very specific problem that dbt Labs identified: agents trained on publicly available data will often use outdated MetricFlow syntax.dbt Labs recently revamped the authoring experience for semantic models. The majority of training data online still uses the old syntax. Without a skill that explicitly teaches the new syntax, agents will confidently write deprecated code — and you won’t realize it until something breaks.Agent Skills can evolve independently of frontier model releases, meaning dbt Labs can push syntax updates to the skill immediately rather than waiting for the next model training cycle.3. Platform Operations SkillsThese skills handle the operational and infrastructure side of working with dbt.Press enter or click to view image in full size4. Migration SkillsThese skills guide agents through the complex, multi-step process of moving dbt projects from one environment to another.Press enter or click to view image in full sizeThe migration skill is particularly valuable because migrations involve many interdependent steps that are easy to get wrong. Encoding the correct sequence into an agent skill means the agent can execute a multi-step migration without losing track of where it is in the process.The Open Standard Behind Agent SkillsAgent Skills are not a dbt-proprietary invention. They are built on an open standard — and in 2026, that standard has been adopted by more than 30 AI agents and platforms.Adopters include:Claude Code (Anthropic, who wrote and donated the open standard)CursorOpenAI CodexGoogle GeminiMicrosoft CopilotDatabricksSnowflake Cortex CodeThis is significant. It means dbt Agent Skills are not tied to a single AI client. Any agent that supports the open standard can load and use them. You write the skill once and it works across your entire AI toolchain — the same portability benefit that MCP gives you at the tool access layer.How dbt Agent Skills Work TechnicallyAt a technical level, dbt Agent Skills are Markdown files. They contain structured natural language instructions that an agent reads before beginning a task.Think of them as a system prompt that travels with the task — a document that tells the agent exactly what the expert-level workflow looks like for a given category of dbt work.For example, the using-dbt-for-analytics-engineering skill might instruct the agent to:Always call get_all_sources before writing any SQL to understand what raw data is availableBuild models one at a time, never batch-creating without validationCall compile after every model to verify Jinja and SQL correctnessWrite schema.yml tests before calling runCall get_model_health after every run to confirm the model landed correctlyThat sequence is obvious to an experienced dbt engineer. It is not obvious to a generalist AI agent. The skill makes it explicit.dbt Labs built a benchmarking library to test different combinations of skills — MCP plus skills, skills alone, no tools at all — to understand exactly how each combination changes agent output. That rigor shows in the quality of the resulting skills.MCP + Agent Skills: The Complete PictureIf you have already read about the dbt MCP server, you might be wondering how Agent Skills fit alongside it. Here is the clearest way to think about it:MCP is the access layer. Agent Skills are the knowledge layer.Press enter or click to view image in full sizeMCP gives an agent the ability to call get_all_models, compile, run, test, and query_metrics. Agent Skills tell it when to call each one, in what order, and what to check before moving to the next step.Used together, they transform a generalist coding assistant into something much closer to a specialized dbt practitioner — one that has read every best practice guide, knows every common failure mode, and never skips a test.How to Set Up dbt Agent SkillsSetting up Agent Skills is straightforward. The repo is open source and the skills are plain Markdown files.Step 1 — Get the Skills Repogit clone https://github.com/dbt-labs/dbt-agent-skills.gitOr access individual skills directly from the GitHub repo at github.com/dbt-labs/dbt-agent-skills.Step 2 — Add Skills to Your AI AgentHow you load a skill depends on which agent you are using.For Claude Code:claude skills add dbt-labs/dbt-agent-skills/analytics-engineeringOr paste the contents of the skill Markdown file directly into your agent’s system prompt or custom instructions.For Cursor:In Cursor, add skills via the .cursor/rules directory in your project root. Create a new .mdc file and paste the skill content:# .cursor/rules/dbt-analytics.mdc---description: dbt analytics engineering best practices---[paste skill content here]For GitHub Copilot / VS Code:Add skills via the Custom Instructions panel in Copilot Chat settings, or as a copilot-instructions.md file in your .github/ folder:# .cursor/rules/dbt-analytics.mdc---description: dbt analytics engineering best practices---[paste skill content here]Step 3 — Pair with the dbt MCP ServerAgent Skills work best when the agent also has MCP access to your dbt project. Without MCP, the agent knows how to work but cannot see your project. With both, it knows the workflow and has access to your real models, metrics, and lineage.If you have not set up the dbt MCP server yet, refer to the local setup guide in the dbt MCP documentation — the full configuration process covers Claude Desktop, Cursor, and VS Code.Step 4 — Test with a Real TaskAfter loading the skill, give the agent a real dbt task and watch how it behaves differently:"Add a new staging model for the payments table and write tests for it."With Agent Skills loaded, you should see the agent:First explore upstream context using available toolsAsk clarifying questions about the source table schemaBuild the model incrementallyWrite schema.yml tests before runningValidate the model after each stepIf the agent jumps straight to building without any of those steps, the skill is not loaded correctly.Which Skills Should You Start With?Not every team needs every skill on day one. Here is a practical starting point:If you are building dbt models daily:Start with using-dbt-for-analytics-engineering. This is the highest-value skill for anyone writing SQL and managing a growing DAG.If you are working with the dbt Semantic Layer:Add building-dbt-semantic-layer immediately. This is essential for avoiding outdated MetricFlow syntax, which is the most common agent failure mode in Semantic Layer work.If your team is migrating to dbt Fusion:Add migrating-to-dbt-fusion. Migration tasks are long, multi-step, and error-prone without a structured playbook.If you want AI to answer business questions:Add answering-natural-language-questions-with-dbt. This skill guides agents to query governed metrics correctly rather than generating ad-hoc SQL against raw tables.Real-World Impact: Three ScenariosScenario 1: A data engineer builds a new martWithout Skills: The agent scaffolds all models at once, misses upstream ref() dependencies, skips tests, uses deprecated config syntax. The engineer spends an hour cleaning up the output.With Skills: The agent explores the DAG first, builds incrementally, writes tests alongside each model, compiles and validates before moving to the next. The engineer reviews clean, testable output.Scenario 2: An analyst queries metricsWithout Skills: The agent writes raw SQL against fct_orders, accidentally excludes the currency conversion macro, returns a revenue number that is 12% off from the official figure.With Skills + MCP: The agent calls list_metrics, finds the governed monthly_revenue metric, uses query_metrics with the correct time grain and dimension, returns a number that exactly matches what the finance team reports. No hallucination. No drift.Scenario 3: A team migrates to dbt FusionWithout Skills: The agent attempts a migration, gets halfway through, loses track of which models have been migrated, uses legacy config patterns for new models, leaves the project in a partially migrated state.With Skills: The agent follows the structured migration playbook step by step, validates each model before proceeding, flags compatibility issues early, and completes the migration cleanly.Common Mistakes to AvoidLoading skills without MCP: Skills guide the agent’s behavior, but without MCP the agent has no project context to work with. The two are most powerful together.Using outdated skill versions: Pin a specific version of the skill when you copy it into your project. Updating the repo regularly ensures you get the latest syntax guidance, especially for fast-moving areas like the Semantic Layer.Loading all skills at once without testing: Start with one skill, validate the agent’s behavior on a real task, then layer in additional skills. Too many instructions at once can create conflicts or confusion.Not customizing for your team: The base skills are a starting point. Experienced teams can fork them and add organization-specific rules — your naming conventions, your testing standards, your deployment process — to make the agent even more precise.Why This Is a Bigger Deal Than It LooksHere is the thing about Agent Skills that most coverage misses.The jump from 56% to 58.5% benchmark accuracy is not the story. The story is that certain tasks that were previously impossible are now reliable.AI agents are increasingly capable at writing code. But data engineering is not just code — it is a series of disciplined, iterative decisions that require domain knowledge, awareness of context, and respect for governance. That is exactly what Agent Skills encode.This is the same insight that made dbt itself successful: the value is not in the SQL execution — it is in the structure, the standards, and the shared understanding of how to work correctly. Agent Skills bring that same philosophy to the AI layer.In 2026, every serious data team will have AI agents working in their stack. The teams whose agents follow best practices and work iteratively will produce reliable, trustworthy output. The teams whose agents have raw tool access but no structured guidance will spend most of their time cleaning up after them.The playbook is available. It is open source. It takes an afternoon to set up.Closing Thoughts & Next Stepsdbt Agent Skills close the gap that MCP alone cannot close. MCP gives agents access to your governed data world. Agent Skills teach them how to be responsible practitioners inside it.Used together, they represent a meaningful shift in what AI-assisted data engineering can actually deliver — not demo-ware, not occasional wins, but reliable, testable, iterative work that meets the standards your team already holds for human contributions.What to do this week:Clone the dbt-agent-skills GitHub repo and explore the available skillsPick the one most relevant to your current work and load it into your AI agent of choiceRun a real task and compare the agent’s behavior with and without the skill loadedIf you are already using the dbt MCP server, pair it with the analytics engineering skill — that combination is the highest-value starting pointConsider forking the repo and adding your team’s own conventions as custom rules on top of the base skills👉If this article clarified how Agent Skills fit into your dbt and AI workflow, share it with your data team — especially anyone who has struggled with AI agents producing unreliable dbt output. If you have already loaded Agent Skills in Cursor, Claude, or Copilot and have real experience to share, drop it in the comments.
