Introduction

Modern geospatial data infrastructure offers multiple approaches to storing and serving vector data. FlatGeoBuf, GeoParquet, Apache Iceberg (with geospatial support), and OGC API-Features represent different paradigms – from file formats to data lake tables to web APIs. This comparison examines their technical data format, access patterns, scalability, and interoperability differences. The newbie here is FlatGeoBuf, which is usable for large-scale cloud analytics and real-time access. The other three, GeoParquet, Iceberg, and OGC API-Features, have been part of our previous, though shorter, publication. At the end of this article, we provide practical recommendations on when FlatGeoBuf is appropriate versus when to prefer GeoParquet, Iceberg, or OGC API-Features.

Data Format and Access Patterns

- FlatGeoBuf (FGB): A single-file binary format based on FlatBuffers, storing a collection of features in a row-oriented layout. Each feature’s geometry and attributes are encoded sequentially, optionally preceded by a packed Hilbert R-tree spatial index for fast bounding-box queries. FGB is cloud-optimized: it supports HTTP range requests so clients can fetch just the bytes for features within a spatial area, rather than downloading the entire file. Access patterns include streaming through the file or performing random reads using the built-in index. Since it’s row-based, reading any feature yields all its attributes and geometry, efficiently retrieving complete features by location. However, it lacks columnar organization and cannot skip unused attributes during reads (no column pruning). FGB files are immutable once written (no in-place edits); they are meant for static or append-only use (appending features at the end is possible if an index is not finalized).
- GeoParquet: Not a new format per se, but a convention for storing geospatial data in Apache Parquet – a columnar table format. Geometries are stored in a Parquet column as Well-Known Binary (WKB), with metadata defining the coordinate reference system and geometry encoding. Because it is columnar, Parquet organizes data into row groups and column chunks, enabling column-level reads and predicate pushdown on attributes. A GeoParquet reader can read a Parquet file (non-geospatial tools will see a binary column for geometry). The access pattern typically involves reading the Parquet file via analytics engines or GIS software: the reader first grabs the file’s footer metadata (which includes schema, column stats, and byte offsets for chunks), then fetches only the needed columns or row groups. This allows efficient scanning of large datasets by loading only relevant parts (e.g., only specific attributes or row groups whose bounding box falls in a query range if such metadata is available). Currently, GeoParquet has no built-in spatial index, so spatial filtering requires scanning geometry data or relying on external strategies (like partitioning the dataset by regions or storing bounding box min/max as separate columns for filtering). The format is read-only/immutable at the file level (a Parquet file cannot be altered in place; updates mean writing a new file). For extensive collections, multiple GeoParquet files can be partitioned by area or attribute, and combined with a metadata catalog (e.g., a top-level \_metadata file or a STAC catalog) for easier access. Accessing specific spatial subsets from a static GeoParquet requires reading at least whole row groups of the file (since there’s no internal R-tree yet), which is less direct than FlatGeoBuf’s index-assisted slicing.
- Apache Iceberg (Geospatial): Apache Iceberg is an open table format for data lakes, layering metadata and partitioning logic on top of files like Parquet/ORC. With recent updates (Iceberg v1.9.0, 2025), it introduces a native geometry column type using WKB encoding, closely aligning with the GeoParquet specification. In effect, Iceberg manages a collection of Parquet files that contain geospatial data. The data format at the file level is typically Parquet (hence also columnar WKB). Still, Iceberg adds rich metadata for access: it tracks all data files in a table, supports partitions (including spatial partition schemes), and records optional spatial statistics (like feature bounding boxes) in metadata for each file to enable query pruning. Accessing data in Iceberg is done via SQL or data frame queries on the table: the query engine consults Iceberg’s metadata to identify which files (or partitions) contain relevant data, then reads those Parquet files. Because Iceberg can partition data by geohash, H3, or spatial tiles, and store min/max coordinates per file, spatial queries can avoid scanning the entire dataset. Access patterns are strongly tied to analytics engines (Spark, Trino, Flink, etc.) and not via direct file I/O by end-users. Iceberg tables support ACID operations so data can be appended or updated transactionally; each write creates new Parquet files and updates the table manifest. This makes Iceberg suited for real-time or continual data ingestion in a lakehouse. Still, it requires running a compute engine or query service – you don’t open an “Iceberg file” directly, you query the table through a supporting engine. In summary, Iceberg provides big-data style access: parallel reads of many files, SQL filters (including on geometry if the engine has spatial SQL functions), and time-travel to different versions. It sacrifices the simplicity of a single-file download in favor of scalable, managed storage and query optimization.
- OGC API-Features: This is not a file format but a web API standard (the modern evolution of WFS, the OGC Web Feature Service standard) for RESTful feature access. Data is exposed as collections of features via HTTP endpoints, typically providing JSON (GeoJSON) representations. However, other formats can be offered (e.g., GML, CSV, or FlatGeoBuf if the server supports it). The access pattern is on-demand queries: a client makes requests with parameters (bounding box, attribute filters, pagination, etc.) and the server returns the matching features. This pattern supports real-time data access – clients can query current data and modify data if the API’s transactional parts are implemented (OGC API-Features has modular parts for create/update/delete operations). Scalability and performance depend on the server’s backend: an implementation might wrap a spatial database, a data lake, or an in-memory store. For example, an OGC API-Features server could be backed by a PostGIS database or an Iceberg/Parquet dataset via a cloud function. The data format internally can be anything, but the API provides interoperability externally through standard request/response patterns. There is no single “OGC API file”; it’s a dynamic service. Access is granular (you request just what you need), which is excellent for interactive applications and integrating with web clients. However, it is not optimized for bulk analytical scans by itself. If a user needs an entire dataset for analysis, pulling it through an API in many requests is less efficient than a direct file download or a query in a data lake environment. In practice, OGC API-Features is complementary to the other formats: one might host data in Parquet/Iceberg or even FlatGeoBuf files on a server, and use an OGC API to serve slices of that data to clients on demand. The key point is that OGC API-Features focuses on interoperability of access (standard HTTP+JSON) rather than defining how data is stored.

Scalability Considerations

- FlatGeoBuf: Designed to handle large static datasets in a single file – it uses 64-bit offsets and counts to avoid size limitations that plague older formats (no 2GB or 4GB barrier). It’s practical to have millions of features or multi-GB files in FGB. The spatial index allows the file to be queried efficiently by region, which means you can work with huge files by retrieving only the needed subsets. For example, a 12 GB nationwide dataset can be kept as one file, and a web map client can fetch just the features within the current viewport via range requests. This eliminates the need to tile or shard the data for read performance, as long as clients do spatial queries. However, FlatGeoBuf’s scalability has limits in other dimensions: being a single file, all data is in one object, so there’s no built-in partitioning by attribute or region (beyond the internal R-tree). Extremely large files (tens of GBs) might still be slow to open if you must read many index pages or if a query spans most of the file. In distributed analytics environments, FGB is not inherently sharded – it doesn’t split work across nodes easily, since typical big data frameworks don’t have native support for it. You’d often end up reading the entire file from one node or implement a custom parallel reader. Also, FlatGeoBuf is best suited for read-heavy, static scenarios; it’s not optimized for high-frequency updates or concurrent writes (rewriting a huge FGB for every update is expensive). In summary, FGB scales well for distributing large datasets to many readers (especially with spatially focused reads). Still, it’s not the go-to for massive parallel processing or constantly changing data.
- GeoParquet: Parquet was built to scale analytic workloads, and GeoParquet inherits that. A single Parquet file can be large (multi-gigabyte) but still efficient to query because of internal indexing (page indexes, column statistics) and compression. More importantly, Parquet is often used in partitioned datasets: instead of one huge file, you can have many Parquet files split by region, time, or other keys. GeoParquet encourages this for extensive geospatial data – e.g., splitting a country-wide dataset into one Parquet per state or tile, and possibly cataloging them with STAC. This approach scales nearly arbitrarily: you can distribute files in cloud storage and use tools like Apache Spark or Dask to read them in parallel. Parquet’s design also allows parallel reads of a single file’s chunks, and multiple files can be read concurrently. However, the current lack of an explicit spatial index means that scaling spatial filtering is a challenge – one might compensate by partitioning data spatially (so queries can read only relevant partitions) or by scanning with parallel threads. GeoParquet’s strength is scaling cloud-native analytics: it integrates well with data lake engines, and even cloud warehouses (BigQuery, Snowflake, etc.) can directly query Parquet files in situ. Conversely, you might hit friction if you try to use GeoParquet in a small-scale or web context. For example, reading a remote Parquet requires pulling metadata and doing random reads, which can be slow over high-latency connections without proper tuning. Remote selective access is an area where Parquet is not yet as straightforward – one community member notes that fetching a spatial subset from a GeoParquet in the browser “requires a software stack from hell” and isn’t worth the effort compared to the simplicity of FlatGeoBuf in that scenario. Thus, GeoParquet scales analytical processing to massive data volumes, but scaling interactive spatial queries (especially without a server) is not its strong suit.
- Apache Iceberg: Iceberg is built for massive scale and multi-user data management. It can handle petabytes of data by distributing it across many files and storage nodes. With Iceberg’s table metadata, queries don’t have to touch every file – only those files that might have relevant data (thanks to partition pruning and metadata filtering). The addition of geospatial partitions and statistics means an Iceberg table can skip loads of data for a spatial query (e.g., if a query asks for features in one city, Iceberg might quickly narrow down to a few Parquet files that cover that city’s tile or geohash). Because Iceberg tables are typically queried via distributed engines, they scale out with compute clusters: hundreds of workers can scan different files in parallel. It’s also designed for high throughput and concurrency – multiple writers can append data (with isolation), and readers always see a consistent snapshot. This makes it suitable for big production environments (e.g., continually updating datasets like telemetry, or large-scale batch processing of geospatial archives). In terms of latency, Iceberg isn’t about microsecond responses – there is overhead in planning the query, reading metadata, etc., so an API or file might be faster for minimal queries. However, Iceberg shines for extensive analytics by avoiding full-table scans and scaling linearly with cluster size. Real-time in Iceberg’s context means near-real-time batch (micro-batches or streaming ingestion can write every few minutes or seconds), not instantaneous per-feature updates. There’s some lag between data arrival and it being committed to the table, but once committed, any query will see the new data. Overall, Iceberg’s scalability is about managing growth: as your geospatial data grows from millions to billions of records, it keeps query performance manageable and enables versioned data lake workflows. The trade-off is complexity and resource cost – you need the infrastructure (e.g., a Spark or Trino cluster or a cloud service) to use Iceberg.
- OGC API-Features: OGC API-Features' scalability depends heavily on its implementation and infrastructure. By itself, the API standard defines how to request data; it doesn’t dictate how to scale. A robust OGC API deployment (for example, a cloud-hosted feature service) can be scaled horizontally: you can run multiple instances behind a load balancer to serve many concurrent requests. The whole system can handle large datasets and high query rates if the backend is a scalable database or data lake. That said, the typical usage of OGC API-Features is to serve relatively granular queries (e.g., “give me features in this bounding box” or “with this property value”) and to page through results. It’s not usually intended for dumping millions of records in a single response (servers often impose limits). For real-time access, an OGC API can be very efficient – clients can repeatedly hit the API for the latest data (e.g., querying a live traffic layer every minute). The overhead per request (HTTP+JSON) is not a big issue if the result set is reasonably small, but it could be significant if clients naively download massive GeoJSON payloads. In a cloud-native setup, one might pair OGC API with cloud functions or distributed query engines to fetch data from Parquet/Iceberg storage on demand. This gives a balance: the data storage scales in the cloud, and the API tier scales by spawning compute as needed. However, designing such an architecture is non-trivial. In simpler terms, OGC API-Features scales access to data for many users, but if each user needs the entire dataset, it scales worse than delivering a bulk file. It excels when users only need slices of the data at a time, which is often the case for interactive mapping or targeted queries. Interoperability-wise, it’s scalable because any client can use the API without custom code (just standard HTTP/JSON), an essential form of scalability (ease of integration).

Interoperability and Ecosystem Support

- FlatGeoBuf: FGB is an open specification (now an OGC Community Standard candidate) and has been integrated into many geospatial tools despite its relative youth. Library support is broad: GDAL (since v3.1) provides read/write, enabling QGIS, MapServer, and other GDAL-based software to use FGB. GeoServer supports FGB as an output format (e.g., WFS/OGC API output), and GeoTools/PostGIS have had integrations or experiments. Notably, web client support exists: projects like OpenLayers, Leaflet, and MapLibre can consume FlatGeoBuf directly via JavaScript libraries or plugins. This means a browser app can fetch an.fgb file (or range-request parts of it) and render features, which is a big interoperability win for web mapping. Because it’s binary, it’s not human-readable, but it’s self-contained: the file includes schema (column names/types), CRS info, etc., so anyone with an FGB reader can interpret the data. Compared to legacy formats, FGB avoids the pitfalls of geospatial file formats (no sidecar files, no character encoding issues, no 32-bit limits) while being faster and smaller than GeoJSON. However, outside the geospatial world, FlatGeoBuf isn’t a mainstream format. General data tools (like Pandas, Excel, and mainstream databases) do notrecognize.fgb. One must use geo-specific libraries to read it. Interoperability is thus high within the geo community, but unlike Parquet, you can’t drop an FGB into a cloud warehouse or big data tool and expect it to load. Another aspect: because FGB is an open spec, it can be implemented in any language; there are already implementations in C++, Python (via GDAL or fgb library), JavaScript, Rust, etc., which lowers barriers to adoption.
- GeoParquet: Being essentially Parquet, GeoParquet benefits from Parquet’s ubiquitous support in data ecosystems. Any tool that knows Parquet (which includes Python Pandas/PyArrow, R’s Arrow package, Spark, Hive, Presto/Trino, BigQuery, Snowflake, etc.) can at least read the tabular portion of a GeoParquet file. If the tool is not geo-aware, the geometry column will appear as binary blobs, but the data isn’t lost; it can be passed along or even operated with custom functions. With the formal GeoParquet 1.0 spec (Sept 2023), there’s now a standard way to declare the geometry encoding and CRS in the Parquet file metadata. This means geo-tools can identify geometry columns. Indeed, GDAL added support in v3.5+ to read/write GeoParquet, so QGIS and other GDAL-based applications can open a \*.parquet file and treat it like a layer. GeoPandas in Python also supports reading/writing GeoParquet out-of-the-box (using [geopandas.read](http://geopandas.read/) \_parquet). Parquet is a de facto standard in enterprise analytics, so using GeoParquet allows geospatial data to blend into non-geo pipelines. For example, a data engineer can join a Parquet of customer locations with another Parquet of sales data in a Spark job, without needing special GIS file handlers. This cross-domain interoperability is a significant advantage. On the Web/JavaScript side, reading Parquet is more complex: there are libraries (Arrow JS can read Parquet), but it’s heavier than reading JSON or even a simple binary format. So GeoParquet isn’t as straightforward as FGB for browser mapping libraries – it typically requires a WASM layer or a server-side component. Additionally, until spatial indices are added to the spec, interoperability with pure GIS applications for fast spatial queries is a bit lacking; many GIS tools expect either to load the whole Parquet or have an external spatial index (some might generate one in memory). In summary, GeoParquet is highly interoperable in data analytics and storage space (including cloud data warehouses) and is increasingly supported by GIS software. It bridges geospatial and big data ecosystems, fulfilling the ideal that a cloud warehouse can directly query geospatial files since they are just Parquet under the hood.
- Apache Iceberg (Geospatial): Iceberg’s interoperability is growing, but is primarily relevant in the big data domain. It’s an Apache project with broad language bindings (Java, Python, etc.), and many query engines support Iceberg tables (e.g., Spark, Flink, Trino, Dremio, Snowflake). The new geometry type in Iceberg 1.9.0 ensures that engines that understand Iceberg can recognize a column as geometric, using WKB encoding for interoperability. If you create an Iceberg table with geometry, a Spark SQL query can filter on that geometry or call a UDF to compute something. Similarly, Dremio or Trino might eventually push down spatial filters by leveraging the stored bounding box stats. Iceberg explicitly followed GeoParquet’s lead on how to store geometry and CRS info, so there’s consistency: an Iceberg table’s Parquet files are essentially GeoParquet-compliant. This compatibility implies that if needed, one could bypass Iceberg and read the raw Parquet files with any GeoParquet-capable tool (though you’d lose the ACID guarantee outside Iceberg). Regarding interoperability with traditional GIS, there’s not yet broad support for connecting QGIS or ArcGIS directly to an Iceberg table. Usually, one would query the Iceberg table via a SQL engine and export the result (maybe as GeoParquet or FGB) for use in a desktop GIS. Some specialized tools or plugins might emerge, but as of now, Iceberg is a bit too “back-end” for most GIS users. That said, Iceberg’s strength is in multi-engine interoperability: you can write data with Spark and read it with Flink or Python, all consistently, which is excellent for complex workflows. It also enables time-travel interoperability – any engine can query a specific table snapshot. Geospatially, this could mean you have a versioned dataset of features that you can query as of last week or last year (which traditional GIS formats can’t do easily). Open-source projects like Havasu and GeoIceberg have been working to extend Iceberg for spatial support, and now much of that is in mainline Iceberg. Overall, Iceberg is very interoperable with modern data lake architectures, but less so with lightweight or legacy GIS workflows. It shines when you integrate it into a cloud stack, not when you need to send a dataset to a colleague (for which you’d convert to a file format like those above).
- OGC API-Features: Interoperability is the raison d’être of OGC API-Features. It provides a standard web interface for any compliant client to access data. This means a wide range of software, from web mapping libraries (Leaflet, OpenLayers via their OGC API clients) to GIS desktop applications (recent QGIS versions, ArcGIS, etc. have some support for OGC API-Features), can consume services without custom connectors. It’s designed around OpenAPI/JSON schemas, so developers can easily understand the endpoints and craft requests. Because it’s an OGC/ISO standard, many vendors implement it, reducing vendor lock-in (one can switch the server, and clients would still work). The format of data returned is typically GeoJSON (which all modern geospatial tools understand), and by using HTTP content negotiation, other formats can be served. For example, a server might offer CSV, HTML, or even a binary like FlatGeoBuf for clients that request it. This flexibility means OGC API can act as a gateway. For example, a client could request Accept: application/x-fgb and get a FlatGeoBuf response if the server supports that, combining API filtering with an efficient transfer format. A practical scenario is an API that serves small queries, such as GeoJSON (for ease of use), but offers a bulk download as a GeoParquet or FGB file for power users. One challenge is that to use OGC API-Features, you need a running service that introduces complexity in sharing data instead of handing over a file. It’s superb for system-to-system interoperability (automated data exchange, live integration in applications), but less so for the ad-hoc “I have a file on disk” use case. In summary, OGC API-Features maximizes interoperability at the access level. It’s the most agnostic to client technology (anything that can make an HTTP request can get data). It doesn’t enforce one data format, which is a strength (flexibility), but also means you rely on underlying formats for actual data representation. Thankfully, those underlying formats can be ones we discussed (GeoJSON, GML, CSV, etc., and potentially new ones like FGB or GeoParquet in the future). This standard is increasingly adopted for modern web GIS and data sharing portals, ensuring that geospatial data providers can reach a broad audience without custom APIs.

Limitations of FlatGeoBuf for Large-Scale Analytics and Real-Time APIs

FlatGeoBuf excels as a lightweight, cloud-friendly container for static spatial data, but it has notable limitations in specific scenarios:

- Limited Utility in Large-Scale Analytics: FlatGeoBuf is row-oriented and lacks features like column pruning or built-in attribute indexing. If you want to perform analytical queries (e.g., “calculate statistics on a population field for all features where type = X”), an FGB file offers no shortcut – you must read through all relevant features. In a big data context (say millions of records on a cluster), there’s no native support for splitting the workload across machines or skipping columns; a Parquet/GeoParquet approach is far superior here. FGB only works well when querying entire features (all attributes) by spatial range, whereas Parquet can retrieve only specific columns and use metadata to skip irrelevant data. In practice, FlatGeoBuf is not optimized for “wide” data, where you have many columns but only care about a few – Parquet would let you avoid I/O on the others. Also, common distributed query engines do not have FlatGeoBuf readers (unlike Parquet), so using FGB in a Hadoop/Spark pipeline would require a custom input format or loading it through GDAL (which can be a bottleneck and not parallelizable by default). FlatGeoBuf also does not internally compress data; while its binary encoding is more compact than text, the files can still be large. Parquet, by contrast, applies columnar compression (often dramatically reducing storage size and I/O). So, for cloud-native analytics on large datasets, especially those involving scanning or aggregating data, FlatGeoBuf is at a disadvantage. It’s better suited for moderately sized datasets or use as an interchange format rather than as the primary storage for an analytic workload.
- Static Nature and Update Overhead: By design, FGB is geared toward static datasets. Updating features in the middle of an FGB file or performing frequent inserts/deletes would require rewriting the file (since there’s no mechanism to adjust the FlatBuffers structures and R-tree on disk in-place dynamically). This makes it ill-suited for real-time data feeds or scenarios with constant edits. For example, if you have an IoT stream of geospatial events, you wouldn’t append each new point to an FGB one by one in production – you’d accumulate and rewrite or use a database/stream processor. Iceberg or a cloud database can handle such continual appends with far less overhead. Additionally, FGB does not have a concept for multi-user editing or ACID transactions; it is simply a disk/cloud storage file.
- Concurrency and Real-Time API Serving: There are some challenges if one tries to use FlatGeoBuf as a backend for an API (as opposed to just a file download). A single FGB file on cloud storage can handle many concurrent read requests (object storage is good at that). Still, each request that needs a different spatial filter might cause redundant reading of the index and different byte ranges, since there’s no query engine coordinating things. An optimized server could load the FGB index into memory and reuse it for queries, but you’ve built a mini spatial server at that point. Moreover, FGB’s strength is spatial filtering; if your API queries involve complex predicates (like “give me all features where attribute X = Y and within this polygon”), the FGB format doesn’t assist with the attribute part – the server would have to scan all candidate features after spatial cut. In contrast, a spatial database or an Iceberg table could use attribute indexes or partition elimination alongside spatial filters. Latency is another consideration: returning a small GeoJSON from a database might be faster than returning an FGB fragment if the overhead of doing range requests and assembling the pieces is high (especially for very small queries). FlatGeoBuf also lacks built-in support for generalized overviews or pyramids for visualization at different scales. If you wanted to use it in a web map for real-time panning/zooming, a single file might be too detailed at small scales (no simplified data for zoomed-out views), whereas an OGC API or vector tile service can serve coarser data for overviews. Essentially, FGB doesn’t handle the cartographic generalization aspect – you’d need to prepare separate data for that or accept heavy loads. Finally, for real-time updates in an API, even if the API serves FGB, you’d have to regenerate or update the FGB file whenever data changes, which could be too slow for frequent changes. An API built on a database can update incrementally.
- Adoption in Big Data Systems: As hinted, mainstream big-data platforms don’t natively support FlatGeoBuf. Cloud warehouses (BigQuery, Athena, etc.) will not query an FGB directly (they do support Parquet, CSV, etc., but a custom binary format is a non-starter for them). While FlatGeoBuf checks many boxes for cloud-native vector, the question is how likely it is to get systems like Snowflake or BigQuery to adopt a new format, as leveraging an existing format (Parquet/Avro) is a more practical path. So, if your goal is to integrate with such platforms, FGB is not the appropriate choice. Similarly, suppose you want to do a Python/Pandas analysis. In that case, there’s no direct FGB loader in Pandas (you’d use GeoPandas with GDAL under the hood, which is fine for moderate data, but not for massive datasets where a Pyarrow Parquet load would be much faster).

## Consigliati da LinkedInIn summary, FlatGeoBuf’s limitations mean that it should be seen as a tool for efficient data exchange and retrieval in specific scenarios rather than a one-size-fits-all solution for big data analytics or dynamic services. It’s excellent for what it was designed for—cloud-optimized static vector data—but other solutions are usually more appropriate when requirements move toward large-scale ad hoc analysis or highly dynamic access.

Role of FlatGeoBuf in Hybrid and Cloud-Native Workflows

Despite the above limitations, FlatGeoBuf can play a meaningful complementary role in modern geospatial workflows, especially in hybrid setups that mix cloud and local or combine different tools:

- Efficient Data Exchange and Distribution: One of FlatGeoBuf’s strongest use cases is as a medium for data transfer. You might have authoritative data in an Iceberg table or a spatial database in a cloud-native stack. Still, when you need to deliver a dataset to users (or to a partner organization, or for offline use), FlatGeoBuf is a convenient choice. It packages everything in one file (schema, CRS, data, index) and can be readily consumed by various clients. For example, a government open data portal could internally store a national dataset in a cloud database, but provide downloads in FlatGeoBuf format for GIS users, combining the ease-of-use of shapefiles with far better performance and no size limits. It also transfers data between systems or microservices in a pipeline. If you need to hand off a large set of features from one service to another (and they can’t directly share a database), serializing to FGB offers a compact, parse-efficient option. Unlike CSV or JSON, it won’t lose data types or precision and is faster to parse (flatbuffers allow near “zero-copy” reads). This can be part of a hybrid workflow where not every component can connect to the data lake – some legacy systems might take file inputs, and FGB can be that interchange format.
- Cloud to Desktop/Offline Bridge: In cloud-native environments, large data often lives in object storage and is processed by cloud engines. However, analysts and GIS specialists sometimes need to pull data down for local exploration or editing. Or consider Open Science scenarios where you need to transfer the results of a scientific platform to serve as input for a follow-up process on another platform. FlatGeoBuf shines in both situations: using cloud APIs or STAC catalogs, one could locate an FGB file for a region of interest, download it, and immediately open it in QGIS or ArcGIS Pro (both of which have added support for FGB via GDAL). GeoParquet could also serve this purpose, but FGB currently has an edge in being more uniformly supported in traditional GIS software (and it’s a single file with no additional index files needed). In the Open Science scenario, FGB works well for cross-platform data exchange, particularly as we frequently experience cross-platform data exchange following a write-once-read-once pattern. For disconnected or edge environments (say, field operations with limited internet), an FGB file on a device can provide a lot of data in a compact form that is queryable without a server. In hybrid cloud architectures, where some processing is cloud-based but some is on-premises, FGB can act as the extract format. E.g., nightly jobs summarize or filter big data into regional FGB files that are then synced to local offices for use in desktop GIS. This leverages the cloud for heavy lifting but uses FlatGeoBuf to deliver results in a ready-to-use way.
- Enhancing Web Applications without Full APIs: FlatGeoBuf can enable a semi-cloud-native pattern where you use static files to achieve dynamic behavior in web apps. As demonstrated by community examples, a large dataset on S3 or CDN can be queried in-browser via FGB’s spatial index, circumventing the need for a custom tile service or feature API. This is attractive for relatively static data layers (e.g., census boundaries, or historical environmental data) that don’t warrant a full server. The application requests the portion of the FGB corresponding to the current map view. In a sense, the file itself acts as a service. This pattern is “cloud-native” in that it relies on cloud storage and client-side computation, rather than monolithic servers – it offloads work to the client and cheap storage reads. That said, it works best when data doesn’t change often (so you can deploy updated FGBs only occasionally) and when clients are expected to handle some logic. It’s a niche but powerful approach for publishing open data interactively without substantial infrastructure.
- Interoperability in Cloud Pipelines: We discussed how FGB isn’t directly pluggable into big data frameworks, but it can still appear in cloud workflows in creative ways. For instance, imagine a serverless function that responds to OGC API requests: it could fetch bytes from an FGB in object storage for the requested bbox and return GeoJSON. Here, FGB is used internally to optimize data access without managing a full database – essentially using the cloud storage + FGB as a poor man’s spatial database. Another scenario is using FGB as an export format from a cloud processing job. For example, a Spark job computes some result, and one of the outputs is writing an FGB (via GDAL or a library) because the next stage or end-user expects that. Many cloud-native geoprocessing frameworks (like Google Earth Engine or AWS Athena GIS) offer exports in CSV or shapefiles; one could envision FlatGeoBuf being added to such services for a more efficient vector export.

In essence, FlatGeoBuf’s role in a cloud-centric world is usually not to be the primary storage of huge, ever-changing data, but rather to augment cloud systems by providing a performant format for exchange, distribution, and client-side access. It helps bridge the gap between the cloud and the edge: cloud systems can generate or host FGB files, and edge clients (browsers, desktops, mobile apps) can directly use them with low friction. This hybrid use (cloud storage + client usage) is likely where FlatGeoBuf will continue to be meaningful.

Practical Recommendations: When to Use Which

Choosing FlatGeoBuf, GeoParquet/Iceberg, or OGC API-Features depends on your use case. Below are recommendations for when each is most appropriate:

- Use FlatGeoBuf when you need to share or deploy static vector data efficiently to various clients, especially when clients will be fetching spatial subsets. FlatGeoBuf is ideal for publishing open datasets (no special server required) and for scenarios where a single file download or access pattern is preferred. If you have a relatively static dataset (or one that updates infrequently in bulk) that is too large for GeoJSON or shapefile, FGB is a great choice to distribute it – consumers will benefit from fast reads and the ability to query by bounding box without extra infrastructure. It’s also a good choice for moving data between environments (cloud to desktop, or between systems) due to its compactness and self-contained nature. In web mapping, use FGB to load large data directly in the browser and allow panning/zooming queries (and the data doesn’t require on-the-fly generalization). Essentially, FGB is preferred for cloud-optimized dissemination of relatively static data. It will shine in download-and-use or browser-based cases where setting up a full API or database is overkill. Remember that consumers need software that supports FGB (which is increasingly common in GIS tools and libraries ).
- Use GeoParquet when you focus on analytics, integration, or very large datasets that need to be stored and queried in a columnar fashion. GeoParquet (a Parquet file with geospatial metadata) is ideal for data science and big-data scenarios: if you plan to run SQL queries, aggregations, or machine learning on the attributes and geometries, Parquet’s performance benefits (column compression, predicate pushdown) will be crucial. It’s the format of choice if you want to leverage cloud data warehouses or frameworks like Spark, for example, storing a billion road segments that you can filter by road type or length without reading everything. Also, choose GeoParquet if your data has many columns or if you anticipate partial access by attributes (e.g., need two columns out of twenty) – Parquet can deliver that efficiently. GeoParquet is also a good archival format for large datasets, as it’s storage-efficient and future-proof (Parquet is an open standard widely supported, and the geospatial metadata ensures self-description). That said, if you intend for users to download and open the data in traditional GIS software directly, check that their tool supports it (support is growing: GDAL/QGIS supports it, but some older tools may not yet). For pure cloud use, this isn’t an issue. In summary, GeoParquet is preferred for cloud-native big data storage and processing, data sharing between geospatial and non-geospatial systems, and cases where analysis (especially attribute-heavy analysis) is prioritized over interactive map display.
- Use Apache Iceberg (with Geo support) when you have large, evolving datasets or need robust data management (transactions, versioning) on data lake storage. Iceberg is the top choice for organizations that treat geospatial data as an enterprise asset requiring ACID guarantees, schema evolution, and multi-engine access. If your use case involves continual updates (append new sensor readings daily/hourly, or regularly edit feature geometries), and you want to be able to query current and past states, Iceberg is appropriate. It’s also ideal when you expect concurrent access by multiple users or jobs without conflicts – e.g., simultaneous writers appending survey data and readers querying analytics. In practical terms, use Iceberg if you already have a data lake stack or want to build a geospatial data lakehouse. A good example is a company that wants to combine imagery metadata, vector features, and tabular data in one lake and needs to run complex analytics or AI pipelines across them: Iceberg provides the table abstraction to organize these. However, Iceberg requires a compatible query engine – it’s not for ad-hoc use by end-users. So you’d choose Iceberg for the back-end infrastructure of a cloud-native system. Downstream, you might still expose the data via an OGC API or export to a file for users; Iceberg is about how you store and govern the data behind the scenes. Another time to prefer Iceberg (or similar table formats) is when your data volume is so high that partitioning and metadata pruning are essential for performance, e.g., nationwide IoT datasets, global telemetry with spatial component, etc., where a monolithic file (FGB or single Parquet) would be unwieldy. In short, pick Iceberg for enterprise-scale data lakes with geospatial data, especially if you need features like time-travel, batch and streaming ingestion, or integration with data warehouse analytics. It’s an investment that pays off for extensive, dynamic data holdings, but probably overkill for small, static datasets.
- Use OGC API-Features when you need to provide on-demand, standard web access to data, especially for interactive applications, integration with web/mobile clients, or when you want to enable queries without exposing the entire dataset. If users should be able to ask, “give me features in this area” or “find features with this property,” and get just those results, an OGC API (Features) is the right tool. It’s perfect for real-time access to the latest data in a database – for example, a live traffic API, or a cadastral data service where updates happen and clients always fetch current info. If your use case involves connecting different systems (e.g., a GIS system pulling data from another organization’s server), OGC API-Features provides a common language that avoids custom integration. Also, it can prevent bulk data transfers and instead deliver data piecemeal as needed (saving bandwidth and allowing clients to work with smaller chunks). From a user perspective, if they only ever need a slice of your data, an API is more convenient than forcing a download of everything. In cloud-native deployments, OGC APIs can be stood up in front of cloud storage or databases to give a REST interface to your data. Choose OGC API when the use case is interactive or integration-focused: web mapping, data dashboards, federated data sharing networks, etc. Do note that if clients require offline or bulk access, you might complement the API with a bulk download option (which could be an FGB or GeoParquet file). So, OGC API-Features is not an exclusive choice; it often works alongside these file formats. But as a guiding rule, prefer OGC API whenever dynamic querying and broad interoperability are top priorities – it offers flexibility that static files cannot, without needing a server environment.

To summarize the recommendations: FlatGeoBuf is best for static data distribution and client-side spatial retrieval, GeoParquet/Iceberg are best for large-scale data analytics and management in cloud environments (Iceberg especially if data is dynamic and needs governance), and OGC API-Features is best for providing standard web service access to your data. In many real-world systems, you’ll use a combination – for example, maintain your master data in Iceberg/Parquet, use an OGC API to serve query results, and offer FlatGeoBuf exports for users who need the whole dataset. By leveraging each technology where it fits, you can build a geospatial data infrastructure that is both performant and user-friendly.
